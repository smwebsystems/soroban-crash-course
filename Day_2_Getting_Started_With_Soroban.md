# Part 1: Introduction to Soroban

## What Is Soroban?

Soroban is Stellar’s smart contract platform.

A smart contract is a program that:

* Lives on the blockchain
* Executes automatically when called
* Enforces rules deterministically

Instead of trusting a person or intermediary, you trust code.

Example use case:

A contract can hold funds and release them only when predefined conditions are satisfied.

---

## Why Rust?

Soroban contracts are written in Rust because:

* Rust provides memory safety
* It prevents common vulnerabilities
* It compiles to WebAssembly (WASM)
* It is efficient and deterministic

Smart contracts must be secure and predictable. Rust supports both.

---

## Why the Soroban SDK?

Smart contracts cannot use Rust’s standard library. Instead, Soroban provides a specialized SDK with blockchain-compatible types and tools:

* `Env` — access to blockchain environment
* `String` — blockchain-safe string type
* `Vec` — Soroban vector type
* `Address` — account representation
* Storage access APIs
* Logging and events

These are necessary because contracts run inside a controlled WASM execution environment.

## Hello World Contract

```rust
#![no_std]
use soroban_sdk::{contract, contractimpl, vec, Env, String, Vec};

#[contract]
pub struct Contract;

#[contractimpl]
impl Contract {
    pub fn hello(env: Env, to: String) -> Vec<String> {
        vec![&env, String::from_str(&env, "Hello"), to]
    }
}

mod test;
```

---

## Detailed Explanation

### `#![no_std]`

Soroban contracts do not use Rust’s standard library.

Why?

Because contracts run in a WASM execution environment with restricted capabilities.
All necessary types must come from `soroban_sdk`.

---

### Importing SDK Components

```rust
use soroban_sdk::{contract, contractimpl, vec, Env, String, Vec};
```

* `contract` — marks this as a Soroban contract
* `contractimpl` — marks callable contract functions
* `vec` — Soroban vector constructor macro
* `Env` — blockchain execution environment
* `String` — Soroban-compatible string
* `Vec` — Soroban-compatible vector

---

### Defining the Contract

```rust
#[contract]
pub struct Contract;
```

This declares a new smart contract.

The struct is empty because the logic lives in the implementation block.

---

### Exposing Contract Methods

```rust
#[contractimpl]
impl Contract {
```

All functions inside this block become callable on-chain.

---

### The `hello` Function

```rust
pub fn hello(env: Env, to: String) -> Vec<String>
```

Parameters:

* `env`: access to blockchain environment
* `to`: a Soroban string

Return type:

* `Vec<String>`

Inside the function:

```rust
vec![&env, String::from_str(&env, "Hello"), to]
```

This:

1. Creates a Soroban vector
2. Allocates it within the environment
3. Returns `["Hello", <name>]`

Example:

Calling `hello("Dev")` returns:

```
["Hello", "Dev"]
```

# Part 3: Testing the Contract

Testing smart contracts is mandatory before deployment.

Soroban provides a built-in test framework.

## Test Code

```rust
#![cfg(test)]

use super::*;
use soroban_sdk::{vec, Env, String};

#[test]
fn test() {
    let env = Env::default();
    let contract_id = env.register(Contract, ());
    let client = ContractClient::new(&env, &contract_id);

    let words = client.hello(&String::from_str(&env, "Dev"));

    assert_eq!(
        words,
        vec![
            &env,
            String::from_str(&env, "Hello"),
            String::from_str(&env, "Dev"),
        ]
    );
}
```

---

## Detailed Explanation

### `Env::default()`

Creates a simulated blockchain environment.

This is not the real network — it’s a local test instance.

---

### `env.register(...)`

Registers the contract inside the test environment.

This simulates deployment.

---

### `ContractClient`

Automatically generated by the `#[contract]` macro.

It provides a type-safe way to call contract functions.

---

### `assert_eq!`

Confirms that:

* The contract returns exactly what we expect
* If it doesn’t, the test fails

Tests ensure contract correctness before deployment.

---

# Section 2: Building a Counter Smart Contract

## Objective

Learn how to store, update, and retrieve persistent data on-chain.

In Section 1, our contract was stateless.
Now we introduce state.

---

## Why State Matters

Real-world contracts must remember things:

* Token balances
* Voting records
* Escrow agreements
* Counters

State is stored on-chain.

In Soroban, this is accessed through the `Env` object.

---

## Counter Contract

```rust
#![no_std]
use soroban_sdk::{contract, contractimpl, Env};

#[contract]
pub struct CounterContract;

#[contractimpl]
impl CounterContract {
    pub fn increment(env: Env) -> u32 {
        let mut count: u32 = env.storage().persistent().get(&"count").unwrap_or(0);
        count += 1;
        env.storage().persistent().set(&"count", &count);
        count
    }

    pub fn get(env: Env) -> u32 {
        env.storage().persistent().get(&"count").unwrap_or(0)
    }
}

mod test;
```

---

## Understanding Persistent Storage

```rust
env.storage().persistent()
```

Soroban provides multiple storage types. Persistent storage:

* Lives across transactions
* Is saved on-chain
* Can be read and updated

---

## How `increment` Works

1. Retrieve current value using key `"count"`
2. If no value exists, default to `0`
3. Increase by 1
4. Save updated value
5. Return new value

This demonstrates full read-modify-write logic.

---

## The `get` Function

Simply retrieves stored value.

If no value exists, returns `0`.

---

# Testing the Counter Contract

```rust
#![cfg(test)]

use super::*;
use soroban_sdk::Env;

#[test]
fn test_counter() {
    let env = Env::default();
    let contract_id = env.register(CounterContract, ());
    let client = CounterContractClient::new(&env, &contract_id);

    assert_eq!(client.get(), 0);
    assert_eq!(client.increment(), 1);
    assert_eq!(client.increment(), 2);
    assert_eq!(client.get(), 2);
}
```

---

## What This Test Proves

* Initial state defaults to 0
* State updates persist between calls
* Storage behaves deterministically
