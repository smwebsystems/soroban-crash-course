# Part 1: Introduction to Soroban

## What Is Soroban?

Soroban is Stellar’s smart contract platform.

A smart contract is a program that:

* Lives on the blockchain
* Executes automatically when called
* Enforces rules deterministically

Instead of trusting a person or intermediary, you trust code.

Example use case:

A contract can hold funds and release them only when predefined conditions are satisfied.

---

## Why Rust?

Soroban contracts are written in Rust because:

* Rust provides memory safety
* It prevents common vulnerabilities
* It compiles to WebAssembly (WASM)
* It is efficient and deterministic

Smart contracts must be secure and predictable. Rust supports both.

---

## Why the Soroban SDK?

Smart contracts cannot use Rust’s standard library. Instead, Soroban provides a specialized SDK with blockchain-compatible types and tools:

* `Env` — access to blockchain environment
* `String` — blockchain-safe string type
* `Vec` — Soroban vector type
* `Address` — account representation
* Storage access APIs
* Logging and events

These are necessary because contracts run inside a controlled WASM execution environment.

## Hello World Contract

```rust
#![no_std]
use soroban_sdk::{contract, contractimpl, vec, Env, String, Vec};

#[contract]
pub struct Contract;

#[contractimpl]
impl Contract {
    pub fn hello(env: Env, to: String) -> Vec<String> {
        vec![&env, String::from_str(&env, "Hello"), to]
    }
}

mod test;
```

---

## Detailed Explanation

### `#![no_std]`

Soroban contracts do not use Rust’s standard library.

Why?

Because contracts run in a WASM execution environment with restricted capabilities.
All necessary types must come from `soroban_sdk`.

---

### Importing SDK Components

```rust
use soroban_sdk::{contract, contractimpl, vec, Env, String, Vec};
```

* `contract` — marks this as a Soroban contract
* `contractimpl` — marks callable contract functions
* `vec` — Soroban vector constructor macro
* `Env` — blockchain execution environment
* `String` — Soroban-compatible string
* `Vec` — Soroban-compatible vector

---

### Defining the Contract

```rust
#[contract]
pub struct Contract;
```

This declares a new smart contract.

The struct is empty because the logic lives in the implementation block.

---

### Exposing Contract Methods

```rust
#[contractimpl]
impl Contract {
```

All functions inside this block become callable on-chain.

---

### The `hello` Function

```rust
pub fn hello(env: Env, to: String) -> Vec<String>
```

Parameters:

* `env`: access to blockchain environment
* `to`: a Soroban string

Return type:

* `Vec<String>`

Inside the function:

```rust
vec![&env, String::from_str(&env, "Hello"), to]
```

This:

1. Creates a Soroban vector
2. Allocates it within the environment
3. Returns `["Hello", <name>]`

Example:

Calling `hello("Dev")` returns:

```
["Hello", "Dev"]
```

# Part 3: Testing the Contract

Testing smart contracts is mandatory before deployment.

Soroban provides a built-in test framework.

## Test Code

```rust
#![cfg(test)]

use super::*;
use soroban_sdk::{vec, Env, String};

#[test]
fn test() {
    let env = Env::default();
    let contract_id = env.register(Contract, ());
    let client = ContractClient::new(&env, &contract_id);

    let words = client.hello(&String::from_str(&env, "Dev"));

    assert_eq!(
        words,
        vec![
            &env,
            String::from_str(&env, "Hello"),
            String::from_str(&env, "Dev"),
        ]
    );
}
```

---

## Detailed Explanation

### `Env::default()`

Creates a simulated blockchain environment.

This is not the real network — it’s a local test instance.

---

### `env.register(...)`

Registers the contract inside the test environment.

This simulates deployment.

---

### `ContractClient`

Automatically generated by the `#[contract]` macro.

It provides a type-safe way to call contract functions.

---

### `assert_eq!`

Confirms that:

* The contract returns exactly what we expect
* If it doesn’t, the test fails

Tests ensure contract correctness before deployment.

---

# Section 2: Building a Counter Smart Contract

## Objective

Learn how to store, update, and retrieve persistent data on-chain.

In Section 1, our contract was **stateless**.

Now we introduce **state**, meaning the contract can **remember information between transactions**.

---

## Why State Matters

Real-world smart contracts must remember data such as:

- Token balances  
- Voting records  
- Escrow agreements  
- Application statistics  
- Counters  

Without state, a contract cannot track history.

State is stored **on-chain** and accessed through Soroban's **Env object**.

---

## Counter Contract

```rust
#![no_std]
use soroban_sdk::{contract, contractimpl, log, symbol_short, Env, Symbol};

const COUNTER: Symbol = symbol_short!("COUNTER");

#[contract]
pub struct IncrementContract;

#[contractimpl]
impl IncrementContract {
    /// Increment increments an internal counter, and returns the value.
    pub fn increment(env: Env) -> u32 {
        let mut count: u32 = env.storage().instance().get(&COUNTER).unwrap_or(0);
        log!(&env, "count: {}", count);

        count += 1;
        env.storage().instance().set(&COUNTER, &count);
        env.storage().instance().extend_ttl(50, 100);

        count
    }
}

mod test;
```

---

## Understanding Storage in Soroban

This contract stores its counter using:

```rust
env.storage().instance()
```

Soroban provides **three storage types**:

| Storage Type | Purpose |
|---------------|--------|
| Instance | Data tied to the contract instance |
| Persistent | Long-term ledger storage |
| Temporary | Short-lived storage |

In this contract we use **instance storage**, which is efficient for storing **contract-specific data like counters or configuration values**.

---

## The Storage Key

```rust
const COUNTER: Symbol = symbol_short!("COUNTER");
```

This defines the **storage key** used to store the counter value.

A **Symbol** is a lightweight identifier used as a key.

Example:

```
Key: "COUNTER"
Value: 5
```

Symbols are efficient because they are optimized for blockchain storage.

---

## How `increment()` Works

The function follows a **read → modify → write** pattern.

### Step 1 — Read Current Value

```rust
let mut count: u32 = env.storage().instance().get(&COUNTER).unwrap_or(0);
```

The contract retrieves the stored value.

If it does not exist, it defaults to **0**.

---

### Step 2 — Log the Current Value

```rust
log!(&env, "count: {}", count);
```

This prints the current counter value to the **contract logs**.

Logs help developers **debug and monitor contract behavior**.

---

### Step 3 — Increment the Counter

```rust
count += 1;
```

The counter value is increased by one.

---

### Step 4 — Store the Updated Value

```rust
env.storage().instance().set(&COUNTER, &count);
```

The updated value is written back to storage.

---

### Step 5 — Extend Storage Lifetime

```rust
env.storage().instance().extend_ttl(50, 100);
```

Soroban storage has a **time-to-live (TTL)**.

This instruction ensures the storage remains active.

Meaning:

- If TTL drops below **50 ledgers**, extend it  
- Set the new TTL to **100 ledgers**

This prevents the storage from expiring.

---

## Why TTL Exists

Soroban uses TTL to prevent **unused storage from bloating the ledger**.

If a contract stops being used, its storage can eventually expire.

Contracts must periodically **extend TTL** if they want data to persist.

---

## What This Contract Demonstrates

This contract introduces several important Soroban concepts:

- Instance storage  
- Storage keys using Symbols  
- Logging with `log!`  
- Storage TTL management  
- Read-modify-write logic  

These are **core building blocks for almost every Soroban smart contract**.
